## Tool Details
- **Name:** Sealbox
- **Repo:** ben-gy/sealbox
- **Category:** encryption-privacy
- **Audience:** Non-technical professionals handling confidential documents
  (accountants, HR, journalists) who need to password-protect a file without
  uploading it anywhere.
- **Stack:** Vanilla TypeScript + Vite 6, Vitest. No runtime dependencies.
- **Browser APIs:** Web Crypto (AES-GCM-256, PBKDF2-SHA-256), Compression Streams
  (gzip), Web Workers, File API/Blob, Web Share, Clipboard, Service Worker (PWA).
- **Worker strategy:** Single dedicated Web Worker; chunked (1 MiB) streaming
  encryption/decryption with determinate byte-level progress + throughput/ETA.

## Privacy Model
- **Protected:** File contents (AES-GCM-256, per-chunk authenticated); original
  filename + MIME type (in an encrypted metadata frame); the password (never
  stored/transmitted, 310k PBKDF2 rounds + per-file salt); integrity (any altered
  byte fails the auth tag).
- **Not protected:** File size (ciphertext ≈ plaintext length); the fact that it's
  a Sealbox file (plaintext `SEALBOX` magic header); a weak password (strength
  meter warns but can't prevent).
- **Trust surface:** Static bundle + TLS for the initial page load only. After
  load, zero network calls during encryption (Service Worker enables full offline
  use). No analytics, cookies, third-party fonts, or telemetry. Password must be
  shared out-of-band.

## Architecture Decisions
- **Vanilla TS, not React:** single input→process→output flow with no complex
  multi-pane state — React would be overhead.
- **Custom authenticated wire format:** 41-byte plaintext header + length-prefixed
  encrypted frames. Per-frame nonce = `baseIv ‖ frameIndex`; frame index also used
  as AES-GCM AAD so frames can't be reordered/dropped/duplicated without the auth
  tag failing. The metadata frame stores the chunk count so truncation is detected.
- **Chunked streaming in a worker:** keeps the UI responsive and allows multi-GB
  files without exhausting memory.
- **Compression Streams gzip (optional):** smaller output for compressible inputs;
  gracefully skipped if the API is unavailable.
- **Compression helper uses the stream reader/writer directly** rather than
  `Blob.stream()` — jsdom (the test env) lacks `Blob.stream()`, and this approach
  works in both Node and browsers.
