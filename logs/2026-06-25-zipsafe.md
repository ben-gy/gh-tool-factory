## Tool Details
- **Name:** zipsafe
- **Repo:** ben-gy/zipsafe
- **Category:** file-tools
- **Audience:** Anyone who needs to open or make a `.zip` without handing private files to a
  web service — office workers on locked-down laptops, freelancers bundling deliverables.
- **Stack:** Vanilla TypeScript + Vite 6, `fflate` ZIP engine, vite-plugin-pwa.
- **Browser APIs:** Web Workers, Transferable ArrayBuffers, File System Access API
  (`showDirectoryPicker`/`showSaveFilePicker`), File/Blob, Drag-and-Drop, Web Share, Service Worker.
- **Worker strategy:** single dedicated Web Worker running all zip/unzip via fflate's
  streaming `Zip`/`Unzip` classes, posting byte-level progress; bytes moved as Transferables.

## Privacy Model
- **Protected:** file contents and filenames never leave the browser; no analytics/cookies/
  third-party assets; extraction paths sanitised against Zip-Slip traversal.
- **Not protected:** ZIP is not encryption (output readable by recipients — points users to
  sealbox); standard ZIP stores names/mtimes in cleartext inside the archive.
- **Trust surface:** the static GitHub Pages bundle + TLS for first load; a Service Worker
  serves everything locally afterwards; no third-party servers/CDNs at runtime.

## Architecture Decisions
- **Vanilla TS over React:** two simple modes over one shared dropzone/result surface — no
  multi-pane state orchestration, so React would be pure overhead.
- **fflate** chosen as the ZIP engine: tiny (~20 KB in the worker chunk), fast, pure-JS
  (no separate WASM file, no COOP/COEP headers needed — important since GitHub Pages can't
  set response headers).
- **Testable engine split:** all ZIP logic lives in `src/zipcore.ts` (pure, callback-based
  progress, runs in Node) so it can be unit-tested directly; `src/worker.ts` is a thin
  postMessage shim over it. Path sanitisation lives in `src/paths.ts` and is tested in isolation.
- **File System Access as enhancement, blob download as the always-available path:** the
  verified happy path doesn't depend on a picker (which needs a user gesture / isn't in
  headless), so the tool degrades cleanly on Firefox/Safari.

## Errors & Resolutions
1. **Control bytes in source regex** — an early `paths.ts` accidentally embedded raw control
   characters in a regex literal; rewrote the file defining the matcher from a string literal
   (`new RegExp('[\\x00-\\x1f\\x7f]', 'g')`) so no control bytes live in source.
2. **TS build errors** — `ExtractedFile` was defined in `zipcore.ts` but imported from
   `./types`; moved it to `types.ts`. `Omit<WorkerRequest,'id'>` collapsed the discriminated
   union's keys; replaced with a distributive omit. fflate's `level` needed a `0..9` union;
   added a clamped cast.
3. **Hidden-banner CSS bug** (caught by screenshot) — `.error-banner { display: flex }`
   overrode the `hidden` attribute, so an empty red banner showed on load. Added a global
   `[hidden] { display: none !important }` rule. Re-verified hidden on reload.
