## Privacy Model
- **Protected:** File contents, column names, row counts, schema, and every query written. There is no upload endpoint in the application. The database is in-memory and dies with the tab; `localStorage` holds interface preferences only.
- **Not protected:** That you visited (anonymous page view); anything on the user's own machine — exports land in Downloads unencrypted; browser extensions with page-read permission.
- **Trust surface:** The site bundle over TLS (including the vendored DuckDB WASM binary), the browser's WASM sandbox, a cookie-less Cloudflare Web Analytics beacon, and `feedback.benrichardson.dev` only on explicit Send.

## Architecture Decisions

**Vanilla TS, not React.** One screen, one piece of real state (the current result). React would have been pure overhead.

**WASM vendored, not CDN.** duckdb-wasm's documented quick-start uses `getJsDelivrBundles()`, which fetches ~35 MB from jsDelivr at runtime. That would put a third party in a position to observe every visitor and would break the offline promise — both fatal to this tool's entire value proposition. All assets are imported through Vite's `?url` so they ship same-origin. Costs bundle size, buys back the privacy narrative.

**Only the `eh` bundle ships.** The build initially included `mvp` as a fallback, doubling the deploy artifact to 73 MB. That fallback was illusory: a browser without WASM exception handling also cannot run the ES2022 module bundle the app compiles to, so it would fail milliseconds later regardless. Dropped `mvp` (artifact → 35 MB) and added an explicit exception-handling feature probe that raises a clear unsupported-browser message instead. `coi` was never included — it needs COOP/COEP headers GitHub Pages cannot set.

**CSV types inferred over the whole file** (`sample_size=-1`) rather than a leading sample. Slower on a large file, but sampling is exactly how a column gets typed `INTEGER` from the first 20k rows and then throws on row 800k — the most common and most baffling CSV failure there is.

**Read-only SQL guard.** `ATTACH`, `COPY`, `INSTALL`, `LOAD` and any `https://`/`s3://` path are rejected before reaching the engine. Not to protect a server — there isn't one — but because DuckDB-Wasm *can* read remote paths, and a query pasted from a forum post should not be able to quietly pull a remote file into a tool that promises nothing leaves the device.

**Windowed grid rather than pagination.** A 20,000 × 30 result is 600,000 cells. Rendering ~30 rows and repositioning on scroll keeps performance flat regardless of result size, and preserves the "it's all just there" feeling that distinguishes this from a paginated viewer.

## Errors & Resolutions

**1. Stale service worker served a different tool.** The first preview load at `localhost:5199` returned *Papershot* — a previous tool's service worker still controlling the shared port. Known trap; unregistered all SWs and cleared caches before every verification pass. Environment issue, not a product bug.

**2. Opening query was a silent no-op.** `openFile` set a `busy` flag cleared only in its `finally`, but `await runQuery()` sat *inside* the `try` — and `runQuery` returns early when `busy`. The file loaded, the schema rendered, and the grid stayed empty. Moved the initial run after the `try/finally`.

**3. Dates rendered as epoch integers.** Arrow returns temporal columns as milliseconds, not `Date` objects, so a date column displayed `1,771,113,600,000`. Added `formatTemporal` / kind-aware `formatValue`; day-precision values render as bare `YYYY-MM-DD`.

**4. Virtualisation defeated on desktop.** `ResultGrid` applied `.grid` to the *same* element already carrying `.grid-host`, so `.grid { height: 100% }` overrode `.grid-host { height: 52vh }`; height resolved to auto, the scroll viewport grew to the full 560,000 px content height, and all 20,000 rows rendered into the DOM. It passed earlier testing only because the run happened at 375 px, where the mobile media query re-asserts the height *after* `.grid` in source order. Moved `.grid` onto an inner wrapper.

**5. Inconsistent number formatting within a column.** Thousands separators were applied per value, so a `Float64` column showed `164,400` directly above `164394.52`. Separators are now reserved for integer-typed columns, where every value gets them.

**6. Export could shadow the source file.** Exporting CSV from a CSV produced the source filename exactly, dropping a *filtered subset* into Downloads under the same name as the complete original. Same-format exports are now suffixed `-filtered`.

**7. A vacuous test caught and re-run.** The header horizontal-sync assertion initially passed against a table narrower than the viewport, where `scrollLeft` clamps to 0 — it proved nothing. Re-ran against a forced 26-column result and confirmed real sync (header `translateX(-300px)`, header and body cells aligned).

**8. Frozen CSS transitions in the preview browser.** The event-log drawer measured as off-screen while `is-open` was set. Cause: the preview tab reports `visibilityState: hidden`, so transitions never advance past their start value. Confirmed by disabling the transition (the drawer snapped correctly to `translate(0)`) and re-ran all drawer geometry checks with transitions neutralised. Environment artifact, not a product bug.
