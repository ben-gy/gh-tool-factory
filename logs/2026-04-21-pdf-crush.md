**Date:** 2026-04-21

## Idea Source
IDEAS.md: "Client-side PDF compression. Accept a PDF, rasterise pages via pdf.js, recompress embedded images with MozJPEG / libwebp2 (via `@jsquash/jpeg` and `@jsquash/webp` WASM), rebuild the PDF with `pdf-lib`. Quality slider, target-size slider, resolution cap, per-page thumbnail grid, before/after size + visual diff. All CPU work in a Web Worker so the main thread stays smooth. Drop zone + determinate progress bar + download + Web Share. Privacy pitch: your documents never leave your device — no uploads, no accounts, no server."

## Tool Details
- **Name:** PDF Crush
- **Repo:** ben-gy/pdf-crush
- **Category:** document-tools
- **Audience:** Office workers, students, anyone with a PDF too large to email or upload to a portal
- **Stack:** Vanilla TypeScript + Vite 6
- **Browser APIs:** Web Workers, Canvas 2D, File API, Web Share API, WebAssembly (MozJPEG via @jsquash/jpeg)
- **Key libraries:** pdfjs-dist 4.x, @jsquash/jpeg, pdf-lib, vite-plugin-wasm, vite-plugin-top-level-await
- **Worker strategy:** Single dedicated worker for MozJPEG JPEG encoding; pdf.js rendering on main thread (async, non-blocking)

## Privacy Model
- **Protected:** Full PDF contents — all processing in-browser, never uploaded
- **Not protected:** GitHub Pages CDN logs page loads (IP, UA, timestamp)
- **Trust surface:** Static bundle on GitHub Pages + TLS; no third-party JS/fonts/trackers

## Architecture Decisions
- Rendered pages on main thread (pdf.js is async, so non-blocking) to avoid nested-worker complexity with pdfjs GlobalWorkerOptions
- Transferred canvas pixel data (ArrayBuffer) to a dedicated worker for MozJPEG encoding — clean separation
- Used `new URL('pdfjs-dist/build/pdf.worker.min.mjs', import.meta.url).href` instead of Vite's `?url` import to avoid TypeScript declaration issues
- Used `vite-plugin-wasm` + `vite-plugin-top-level-await` for clean WASM bundling in worker context
- Light theme (white/blue) chosen for consumer audience vs Dropwell's dark developer theme
- Three TypeScript fixes needed post-first-build: worker postMessage signature, Uint8Array<ArrayBufferLike> Blob typing, `?url` import type

## Errors & Resolutions
1. `TS2769` on `self.postMessage(out, [jpegBuffer])` in worker — DOM lib types `self.postMessage` as `Window.postMessage`; fixed by casting `self as DedicatedWorkerGlobalScope`
2. `TS2307` on `pdfjs-dist/build/pdf.worker.min.mjs?url` — Vite's `?url` virtual module not typed for external packages; switched to `new URL('...', import.meta.url).href`
3. `TS2322` on `new Blob([outBytes])` — TypeScript 5.7 makes `Uint8Array` generic, `BlobPart` requires concrete `ArrayBuffer`; fixed by wrapping in `new Uint8Array(outBytes)` to force a copy with a concrete type
