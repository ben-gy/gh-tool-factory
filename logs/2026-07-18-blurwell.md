Searched the client-side face-blur space and found it genuinely crowded: iloveimg, blur-face.com, picdefacer, Filator, BlurShot and others all advertise in-browser processing, several honestly. Video was therefore the reason to build, not a bonus feature. The [[mediabunny-webcodecs-video]] memory from the vidwell build made the video pipeline a known quantity rather than a gamble.

**Expansion check:** considered and rejected as an expansion of existing tools. `metascrub` strips metadata (not pixels), `bgwipe` uses ML on images but for background removal, `veilpix` hides data inside images. Blurwell is a different problem domain (identity redaction), a different modality (video as the headline), and a different central API (MediaPipe FaceDetector). Built as a new tool.

## Privacy Model

- **Protected:** files never leave the device (no upload endpoint; runs fully offline); detection is local against a vendored model; **no face recognition or embeddings, ever** â€” the model returns rectangles only; redaction is burned into exported pixels; EXIF/GPS dropped from image exports as a by-product of re-encoding; audio optionally stripped.
- **Not protected:** detection misses profile views, distant faces and heavy motion blur â€” exports must be reviewed; the original file on disk is deliberately untouched so the user must share the export; blurring a face doesn't hide clothing, location, vehicle or gait; weak pixelation is partially reversible (hence heavy blur as default); GitHub Pages sees the IP on page load.
- **Trust surface:** the GitHub Pages bundle built from the public repo, the TLS chain, and the cookie-less Cloudflare Web Analytics beacon. No other third-party requests â€” model, WASM and fonts are all same-origin.

## Architecture Decisions

**Two-pass video, and why it isn't over-engineering.** A detector run independently per frame will miss faces on some frames; each miss is an un-blurred frame of a face in the published file. A single forward pass can only extend coverage *forwards* from a detection, leaving the frames immediately before a face is first picked up exposed. Scanning the whole clip first (decode-only, ~15 Hz sampling) and then rendering against a complete timeline makes the persistence window **bidirectional**. The cost is a second decode pass; decode is cheap next to encode, and it also enables an honest "N faces found" count before rendering starts. Padding (18% default) and overlap-merging complete the over-cover bias.

**Vendoring the model and WASM (~11.4 MB) instead of using Google's CDN.** A CDN fetch would leak that the user is anonymising something and would break offline operation â€” which is the single most checkable privacy claim the tool makes. The size is comparable to scribewell/bgwipe and is cached after first load. The service worker deliberately does *not* precache the 11 MB WASM (per [[jsquash-encoder-gotchas]]); it's cached on first use so install isn't blocked by a huge download.

**Merge criterion changed from IoU to containment ratio.** Originally boxes were merged on IoU > 0.1. A unit test for transitive merging (A overlaps B, B overlaps C) failed, and the reason turned out to be substantive rather than a bad test: IoU falls off sharply when boxes differ in size, so a small face box sitting almost entirely inside a larger merged region scores low and stays separate. Switched to `intersection / area(smaller)` with a 0.2 threshold.

**MediaPipe over TF.js BlazeFace.** MediaPipe's runtime is much heavier (11 MB vs ~1.5 MB) but is actively maintained, has an official worker sample, and takes `modelAssetBuffer` so the model can be vendored and fed in directly. Reliability was worth the bytes for a one-shot autonomous build.

## Errors & Resolutions

1. **`ModuleFactory not set.`** â€” face detection failed immediately. Diagnosed by confirming all assets served 200, then inspecting `vision_wasm_internal.js`: it's a UMD script that defines `ModuleFactory` as a plain global with no ESM export. In an ES module worker it's pulled in via `import()`, lands in module scope, and the runtime never finds it. **Fix:** switched to a classic worker (`worker.format: 'iife'`, `new Worker(url)` without `{type:'module'}`), where `importScripts()` exists. mediabunny bundled cleanly into the iife worker. Documented in the README and in a `vite.config.ts` comment so it doesn't get "tidied up" into a module worker later.

2. **`hidden` attribute silently ignored on three panels.** The progress panel and error banner were rendering permanently below the fold, and video-only controls showed for images. Cause: author rules like `.progress-panel { display: flex }` beat the UA stylesheet's `[hidden] { display: none }`. **Fix:** explicit `[hidden] { display: none !important; }` early in the stylesheet. Caught only because the dry-run asserted on computed styles rather than the `hidden` property.

3. **`INVALID_ARGUMENT: Packet timestamp mismatch` on video scan.** MediaPipe's VIDEO mode requires strictly increasing timestamps. Sampling on a fixed 15 Hz grid over a sparse video (the test clip had 4 frames over 3.6 s, because background-tab timer throttling limited MediaRecorder) makes `canvasesAtTimestamps` resolve consecutive sample points to the *same* decoded frame, repeating its PTS. **Fix:** skip repeated frames outright (also avoids wasted inference) and force a monotonic detector clock as a second guard. The accidentally-sparse test video was what exposed this â€” a dense video would have hidden it until a real user loaded a slideshow or a low-fps screen capture.

4. Minor: floating event-log pill overlapped the footer links at 375px â€” added bottom padding to the footer on small screens.
