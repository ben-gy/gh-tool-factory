# Build Log: Glyphcut
**Date:** 2026-07-21
**Status:** deployed

## Idea Source

Researched. `IDEAS.md` contained no queued ideas (only the criteria bullets), so I looked for a
gap in the existing catalogue. The 24 shipped tools already cover PDF, images, media, crypto,
archives and CSV; **fonts were an untouched domain**.

The deciding factor was the privacy narrative, which is unusually concrete here. Every competing
online subsetter — Transfonter, Font Squirrel's Webfont Generator, and the long tail of
"convert ttf to woff2" sites — requires uploading the font file to a server. Commercial font
EULAs routinely prohibit transferring the font software to a third party, so that upload is a
plain licence breach for a large share of real users; and for a bespoke or unreleased typeface the
file *is* the confidential asset. This is the rare case where "runs in your browser" is not a
nice-to-have but the entire reason to choose the tool.

Googleable queries imagined: *"subset font online"*, *"ttf to woff2 converter"*,
*"reduce web font file size"*, *"convert font without uploading"*.

## Feasibility probe (before committing to the design)

Rather than assume the libraries worked, I proved the pipeline first in Node:

- `harfbuzzjs@1.4.0` ships `harfbuzz-subset.wasm` (507 KB) with all 39 needed exports
  (`hb_subset_or_fail`, `hb_subset_input_unicode_set`, `hb_set_add`, …) but **no JS wrapper** for
  the subsetter — so the bindings had to be hand-written.
- Probe result on a real 750 KB, 3,381-glyph TrueType font: **750,984 B → 17,260 B subset →
  8,576 B WOFF2**, and the WOFF2 decompressed back to a valid font.

That probe also caught two design decisions early:

1. **`woff2-encoder` over `wawoff2`.** `wawoff2` is a CommonJS Emscripten build that `require`s
   `fs`/`path` — awkward under Vite. `woff2-encoder` is ESM-only, browser-first, bundles its own
   WASM, and does both compress *and* decompress (needed for `.woff2` input).
2. **Dropped `opentype.js` entirely.** It returned `undefined` for Arial's family name (its `.en`
   name lookup missed the platform encoding) and it parses full glyph outlines I never use. I
   wrote a first-party SFNT reader instead — which correctly reads that same family name.

## Tool Details
- **Name:** Glyphcut
- **Repo:** ben-gy/glyphcut
- **Category:** developer-utilities (indexed under `files`)
- **Audience:** Front-end developers and designers optimising web font delivery
- **Stack:** Vite 6 + vanilla TypeScript + Vitest
- **Browser APIs:** WebAssembly (HarfBuzz `hb-subset`, WOFF2/Brotli), Web Workers, Transferable
  ArrayBuffer, CSS Font Loading API (`FontFace`), Compression Streams, File API, Drag and Drop,
  Clipboard, Web Share, Service Worker
- **Worker strategy:** one dedicated ES module worker owning both WASM modules. Font bytes are
  transferred in once and held worker-side for the session, so re-cutting with a different charset
  never re-reads the file and the main thread never touches font data.

## Privacy Model
- **Protected:** the font file and the pasted subsetting text never leave the device; metadata is
  parsed locally; both WASM binaries are vendored same-origin (no CDN), so no third-party origin is
  contacted at runtime; works fully offline after first load.
- **Not protected:** GitHub Pages logs the page request; Cloudflare Web Analytics counts an
  anonymous page view; feedback is sent only when deliberately submitted. Critically, the UI states
  plainly that Glyphcut **cannot check your font licence** — it removes the technical need to
  upload, and the `fsType` bits it displays are the font's own embedding declaration, not the
  licence.
- **Trust surface:** the static bundle over TLS, plus the two vendored WASM binaries pinned in the
  repo.

## Architecture Decisions

**Hand-written raw HarfBuzz WASM bindings.** No wrapper exists for the subsetter in `harfbuzzjs`.
The binding discipline that matters: every HarfBuzz call can grow the WASM heap, which detaches
existing typed-array views, so `heap()` rebuilds the view on every access and no view is cached
across a call. All handles are released in a `finally` block.

**Kept the `name` table in the output.** `hb-subset` drops most name records by default, leaving
the cut font anonymous. Inverting the `HB_SUBSET_SETS_NAME_ID` set keeps them, so the output still
identifies its family, style and licence — and the live preview shows the right family name.

**First-party SFNT reader over a full font parser.** `src/sfnt.ts` understands exactly six tables
and *skips what it doesn't recognise*. This is more robust, not less: a complete parser must
understand every table it meets and throws on unusual ones, whereas a font Glyphcut cannot fully
describe still loads and still subsets.

**WOFF 1.0 via Compression Streams.** `.woff` tables are zlib-compressed, which
`DecompressionStream('deflate')` inflates natively — a whole input format supported with no extra
dependency.

**The specimen renders from the output file.** The preview loads the subsetted WOFF2 back through
`FontFace`. For a skeptical audience, showing the real artifact beats showing a percentage.

## Bugs Caught & Fixed During Verification

1. **Control characters reported as "missing".** The Latin preset spans `U+0000-00FF`, which sweeps
   in C0 control codepoints that no font maps. The result panel therefore announced *"125 requested
   characters are not in this font — U+0000 U+0001 U+0002 …"*, which made a perfectly good 98.5%
   subset look broken. Added `isReportableCodepoint()` to filter C0/C1 controls, soft hyphen,
   zero-width and bidi marks, BOM and the replacement character. The warning now lists 50 genuinely
   missing *printable* characters (U+2010 hyphen, U+2023 bullet, U+2031 per-ten-thousand sign) —
   which is real, actionable information.
2. **`Omit<WorkerRequest, 'id'>` collapsed the message union.** A plain `Omit` over a union reduces
   it to the keys every member shares, dropping `fileName` and `fontId` and failing the build.
   Replaced with a distributive `DistributiveOmit`.
3. **`const enum` broke under `isolatedModules`.** Converted to a plain `as const` object.

## Non-bug: hidden-tab CSS transitions

The mobile log-drawer initially appeared not to open — `is-open` was set but the computed transform
stayed pinned at `translateX(100%)`, even after waiting well past the 250 ms transition, and even
when forced via an inline style. Cause: the preview browser reports
`document.visibilityState === 'hidden'`, and **CSS transitions do not advance in a non-painting
tab** (the same root cause as the known `requestAnimationFrame` hang, which is worth remembering
applies to transitions too). Setting `transition: none` proved the CSS correct — the drawer snapped
to `left: 0` with the close button on-screen at 329–359 px and passing `elementFromPoint`. No code
change was needed.
