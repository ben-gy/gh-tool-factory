Researched. `IDEAS.md` contained no queued ideas (no lines starting with `-`), so I selected a

veilpix, metascrub), files (zipsafe), images (bgwipe), PDF (pdf-crush) and P2P (dropwell) —

modality with a strong, distinct browser-API stack and a highly Googleable problem
("convert video to gif without uploading", "private mp4 to gif", "video to gif no watermark").
Passes all five gates: zero backend, non-trivial browser APIs (video decode + canvas + workers),
honest privacy story (the clip never leaves the device), real Googleable problem, not a duplicate.

## Privacy Model
- **Protected:** the source video, every sampled frame, and the finished GIF — all decoded,
  resized and encoded in the tab; the GIF hits disk only on explicit download.
- **Not protected:** the initial static-page load (GitHub Pages/CDN logs it); what the user does
  with the GIF afterwards. The video content itself is never transmitted.
- **Trust surface:** the hash-pinned site bundle, the TLS chain to GitHub Pages, and nothing
  else — a strict CSP (`connect-src 'self'`) blocks any runtime network call; no third-party
  scripts/fonts/analytics.

## Architecture Decisions
- **Vanilla TS over React:** the tool is a linear input→process→output workflow (idle → editor →
  result), not a multi-pane state app. Vanilla keeps the bundle ~26 kB JS + 13 kB CSS.
- **Seek-based frame extraction** (set `currentTime`, await `seeked`, `drawImage`, `getImageData`)
  rather than WebCodecs demuxing: works with any browser-playable container, no fragile MP4 box
  parsing, and supports precise trim ranges. The CPU-heavy quantize/encode runs in a worker.
- **gifenc** for encoding: small, fast, pure-JS (no separate WASM file to serve), runs unchanged
  in both the worker and the Vitest suite, with per-frame palettes for better colour fidelity.
- **"Try a demo GIF" synthesizes frames directly** (canvas → RGBA) and runs them through the real
  worker, instead of the original MediaRecorder approach. Rationale discovered during browser
  verification: MediaRecorder-produced WebM reports a broken duration (~0.001 s) and rAF stalls on
  backgrounded tabs / in headless preview. Synthesizing frames is more robust (no codec/duration
  dependency), works in every environment, and exercises the genuine encode pipeline.
- **PWA via vite-plugin-pwa** (generateSW, autoUpdate) for offline support after first load.

## Errors & Resolutions
- **TS build errors (first pass):** gifenc ships no types (added `src/gifenc.d.ts`); an unused
  `openModal` import (removed); `Blob([Uint8Array])` rejected under TS 5.7's `ArrayBufferLike`

  verification (and improved for real users) by switching the "try it" button to synthesize frames
  directly and encode them through the real worker — the encode pipeline is now fully verified
  in-browser; the seek-extract front half remains environment-limited but is standard code with
