product geometry). Competitor check confirmed the space is served mostly by upload-first sites.

- **Category:** 3d-tools (indexed under `files`)
- **Audience:** 3D-printing hobbyists and makers; product designers and engineers sanity-checking

- **Stack:** Vite 6 + vanilla TypeScript + Vitest. One runtime dependency: `fflate`
- **Browser APIs:** WebGL2, Web Workers, Transferable ArrayBuffer, Pointer Events, File System
  Access, Drag and Drop / File API, Web Share (files), Clipboard, ResizeObserver, Service Worker
- **Worker strategy:** single dedicated module worker owning parse → analyse → export, posting
  determinate progress. The worker keeps the authoritative geometry for the session; the main
  thread holds a copy only to feed the GPU, so exports need no round trip.
- **Bundle:** 47.9 KB JS (17.4 KB gzipped), 16.4 KB CSS

## Privacy Model
- **Protected:** the model file never leaves the device — no upload endpoint, no backend. Parse,
  analyse and export all run in a Web Worker in the tab. No filename, dimension or
  geometry-derived value is transmitted. Fully offline after first load. No cookies, no
  fingerprinting, no third-party fonts.
- **Not protected:** GitHub Pages sees the page request (IP, user agent). A file the user exports
  and then chooses to share goes wherever they send it. Explicitly not DRM and not a licence
  checker.
- **Trust surface:** the static bundle and its TLS chain; the cookie-less Cloudflare Web Analytics
  beacon; the feedback endpoint (which receives nothing unless the form is opened and sent). A CSP
  meta tag restricts the page to `'self'` plus those two origins.

## Architecture Decisions

**No three.js.** The viewport needs flat-shaded triangles, coloured line lists and a grid. A
scene graph would be an order of magnitude more shipped code for no gain, and the single most
important feature — drawing broken edges in red over the surface — is a second draw call rather
than a material system. The hand-written renderer is ~300 lines and the whole app is 48 KB.

Two renderer choices follow directly from "the models this tool loads are the broken ones":
- **No normal buffer.** The face normal is recovered in the fragment shader from screen-space
  derivatives of the world position — which *is* flat shading, correct for a faceted print mesh,
  and halves the geometry uploaded to the GPU.
- **Culling off, two-sided lighting.** A mesh with inverted normals would otherwise render
  inside-out or invisible, and reporting inverted normals is part of the job.

**Hand-written parsers for all four formats.** Not purism: a Web Worker has no `DOMParser`, and
every off-the-shelf 3MF reader assumes one. Writing them keeps the entire pipeline inside the
worker, where 300 MB meshes belong.

**Welding is the whole game.** STL stores each triangle's corners independently, so a solid cube
arrives as 36 loose points and *every* edge reads as a hole until coincident corners are merged.
The spatial hash uses 1e-5 of the bbox diagonal (a micron on a 100 mm part). The subtle part is
cell boundaries — two corners a nanometre apart can land in different cells — so a corner within
a quarter-cell of a boundary also probes the neighbour on that axis. Two corners that straddle a
boundary are both near it by definition, so they always meet, while the overwhelming majority
still cost one lookup.

**Volume via the divergence theorem**, not voxels: correct wherever the origin sits, and the sign
reveals inside-out models for free.

**Material estimate models walls explicitly** (surface area × wall thickness for the shell, infill
applied only to the remainder). Volume × density × infill% is wrong for most parts and badly
wrong for small ones, where the perimeters *are* the part.

**Analysis runs once.** Unit and scale changes re-derive display values by s¹/s²/s³ rather than
re-analysing, so the scale controls are instant on huge meshes.

## Errors & Resolutions

**1. `tsc` failed on an unused constant.** `PLY_SIZES` was left over from an earlier design where
the binary PLY reader advanced a cursor by table lookup; the final reader advances per type
inside its own `read()`. Deleted.

**2. The report panel stretched the layout instead of scrolling (found in browser verification).**
The workspace grid declared only `grid-template-columns`, so the implicit row was `auto` —
sized by its tallest content. The long report panel therefore sized the row (1505 px measured)
rather than scrolling inside it, and the canvas, which fills that row, grew with it and rendered
the model at a wildly wrong aspect ratio — it looked like a camera stuck at extreme zoom. Fixed
with `grid-template-rows: minmax(0, 1fr)` plus a fixed-height desktop application shell
(`#app { height: 100dvh; overflow: hidden }` above 900 px), leaving mobile in normal document
flow. This would have shipped visibly broken on every desktop window.

**3. The renderer drew into a 0×0 canvas.** A canvas that has not been laid out yet measures
zero; the original `resize()` clamped that to 1×1 and drew anyway, producing a one-pixel image
that the browser stretches across the full element the moment it becomes visible. `resize()` now
returns false and `render()` declines to draw until there is a real box; the ResizeObserver and a
new `visibilitychange` listener ask for the frame again. (Investigating this is what surfaced
bug 2 — the initial hypothesis was wrong, and measuring the canvas is what found the real cause.)

**4. Stale service worker on localhost:5199** — the known shared-preview-origin trap. Unregistered
the previous tool's SW and cleared its caches before each verification pass.

**5. A mis-clicked coordinate on the production check** briefly looked like a hung worker. The
"Analysing… / Reading the file" text was the static default markup, not live state — the click
had missed the button. Confirmed by probing the deployed worker directly (it replied correctly),
then re-clicked by element reference and the full flow ran. No code change needed.
