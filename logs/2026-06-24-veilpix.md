## Idea Source
Researched. `IDEAS.md` contained no queued ideas (only the explainer header), so a
new idea was chosen. Surveyed the existing registry — dropwell (P2P transfer),
pdf-crush (PDF compression), bgwipe (ML background removal), metascrub (metadata
stripping), sealbox (password file encryption) — and picked a genuinely new
problem domain: **steganography**. No existing tool hides data inside images, and
the central browser-API pillar (Canvas `ImageData` LSB manipulation + Web Crypto)
is distinct. Imagined Google query: "hide encrypted message in image online" /
"steganography tool no upload".

## Tool Details
- **Name:** Veilpix
- **Repo:** ben-gy/veilpix
- **Category:** encryption-privacy
- **Audience:** privacy-conscious / technical users (journalists, activists, security hobbyists)
- **Stack:** Vite 6 + vanilla TypeScript, no runtime dependencies
- **Browser APIs:** Web Crypto (AES-GCM-256 + PBKDF2), Canvas 2D / ImageData, createImageBitmap, Web Workers + transferable ArrayBuffer, Web Share, Clipboard, Service Worker (PWA)
- **Worker strategy:** single dedicated Web Worker for crypto + LSB embed/extract, phase-level progress, zero-copy pixel transfer

## Privacy Model
- **Protected:** secret contents (AES-GCM-256), secret filename (sealed in ciphertext), cover image (decoded/re-encoded locally, never uploaded), passphrase (never stored/sent)
- **Not protected:** the *presence* of hidden data (LSB stego is detectable by steganalysis); rough payload-size upper bound; durability through recompression (JPEG conversion destroys the payload)
- **Trust surface:** static site bundle + TLS to GitHub Pages (page load only); passphrase shared out-of-band; no server, analytics, fonts, or network calls at runtime (CSP `connect-src 'self'`)

## Architecture Decisions
- **Vanilla TS over React:** single-screen input→process→output workflow with two
  modes (Hide/Reveal); no complex interacting panes, so React overhead wasn't
  justified. Matches the dropwell/sealbox family.
- **Pure, DOM-free core (`crypto.ts`, `stego.ts`, `pipeline.ts`):** all crypto +
  container framing + LSB bit I/O is plain byte-array code, so the worker, main
  thread, and Vitest share exactly one code path and the logic is exhaustively
  unit-testable without a browser.
- **Opaque-pixels-only embedding:** the canvas alpha round-trip pre-multiplies and
  would corrupt LSBs of partially-transparent pixels, so only alpha=255 pixels
  carry data. Output is always lossless PNG.
- **No external libs:** Web Crypto + Canvas cover everything; keeps the bundle tiny
  (~26 KB JS gzip 8.8 KB, worker 4.2 KB) and the trust surface minimal.

## Errors & Resolutions
- **TS build errors (first pass):** unused `openModal`/`onChange`; `new ImageData(Uint8ClampedArray<ArrayBufferLike>)` generic mismatch; `HTMLElement` vs `HTMLButtonElement` for action buttons; `Omit<WorkerRequest,'id'>` collapsing union keys (lost `isFile`); `import.meta.env` untyped. Resolved by: removing unused symbols; switching to `ctx.createImageData().data.set(pixels)`; casting action buttons to `HTMLButtonElement`; typing `runJob` as `Omit<EmbedRequest,'id'> | Omit<ExtractRequest,'id'>`; adding `src/vite-env.d.ts`.
- **Duplicate modal overlays (found in browser dry-run):** `initModalTriggers()` ran on every `render()`, stacking click listeners so one click opened 4 overlays. Fixed by making it idempotent via a `data-modal-wired` flag.
- **Mobile horizontal overflow (found in browser dry-run):** the hero tagline didn't wrap and the trust banner (a `<button>`) inherited `white-space: nowrap` from the global button rule, clipping its text. Fixed by stacking the hero on ≤640px and setting `white-space: normal` on `.trust-banner`.
