# Build Log: Vidwell
**Date:** 2026-07-18
**Status:** deployed

## Idea Source
Researched. IDEAS.md was empty, so I mapped the existing catalog and found a clear
gap: the catalog covered images, PDFs, audio (trim/transcribe) and media (screen
record, video→GIF) but had **no video compressor/trimmer** — one of the single
most-Googled browser tasks ("compress video online no upload"). Confirmed against
`registry.json` and `gh repo list ben-gy` that no existing tool does this. Chose
**WebCodecs** as the ambitious central API (no server, and — unlike ffmpeg.wasm —
no COOP/COEP headers, which GitHub Pages can't set). Discovered `mediabunny`
(v1.50.8), a modern all-in-one WebCodecs demux→transcode→mux library by the author
of mp4-muxer, which de-risked the build substantially.

## Tool Details
- **Name:** Vidwell
- **Repo:** ben-gy/vidwell
- **Category:** media-tools
- **Audience:** Anyone who needs a big video to be a small video right now (email, upload caps, sending a personal clip) without uploading it to a server.
- **Stack:** Vite 6 + vanilla TypeScript
- **Browser APIs:** WebCodecs (VideoEncoder/VideoDecoder/AudioEncoder), Web Workers + transferable ArrayBuffer, HTMLVideoElement (preview/trim), Web Share, Service Worker (PWA)
- **Worker strategy:** single dedicated Web Worker owns the mediabunny `Conversion`; streams determinate progress; returns the MP4 as a transferable ArrayBuffer; `conversion.cancel()` on abort.

## Privacy Model
- **Protected:** source video, decoded frames and the compressed MP4 never leave the device; no upload endpoint; no account/cookies/watermark; works offline.
- **Not protected:** output is an ordinary unencrypted MP4; compression is lossy by design; decode/encode support depends on the browser's WebCodecs.
- **Trust surface:** hash-pinned static bundle + TLS to GitHub Pages; native WebCodecs + bundled mediabunny/AAC WASM; cookie-less Cloudflare Web Analytics page-view beacon.

## Architecture Decisions
- **Vanilla TS** over React — single workspace, no component tree.
- **mediabunny high-level `Conversion` API** rather than a bespoke WebCodecs pipeline — it handles demux (MP4/MOV/WebM/MKV), H.264 encode (`codec: 'avc'`), resize (`width/height/fit`), frame-rate cap, trim, and audio stream-copy/AAC re-encode, all tested.
- **`@mediabunny/aac-encoder`** registered so AAC re-encode works cross-container (e.g. WebM/Opus → MP4) even where the browser's WebCodecs lacks AAC encode.
- **Bitrate model:** bits-per-pixel × pixels × fps, per quality preset — resolution-aware and gives a concrete number to both drive the encoder and show a live size estimate.
- **Resolution caps** applied to the *shorter* side (orientation-agnostic, downscale-only, even dimensions).
- All pure maths (dimensions, bitrate, size estimate, trim normalisation, filename) isolated in `compress.ts`/`format.ts` and unit-tested.

## Errors & Resolutions
- **mediabunny `fit` requirement:** the first real dry-run failed with "when both options.video.width and options.video.height are provided, options.video.fit must also be provided." Caught cleanly by the error path (message + retry shown). Fixed by adding `fit: 'fill'` when both dimensions are set (target dims are computed from the source aspect, so 'fill' resizes exactly without letterbox bars). Rebuilt and re-ran → success.
- **TS strictness:** `output.target.buffer` typed as ArrayBufferLike and an unused `closeModal` import — resolved by copying any view into a fresh ArrayBuffer and trimming the import.
