## Tool Details
- **Name:** bgwipe
- **Repo:** ben-gy/bgwipe
- **Category:** image-tools
- **Audience:** Non-technical — ecommerce sellers (Etsy/eBay), social-media managers,
  designers, parents. Light theme, spacious layout, no terminal aesthetic.
- **Stack:** Vanilla TypeScript + Vite 6 (no framework)
- **Browser APIs:** @huggingface/transformers v3, WebGPU (with WASM fallback),
  Web Workers (ES modules), OffscreenCanvas, Cache API, Clipboard API, Web Share API,
  File API / DataTransfer, paste ingest
- **Worker strategy:** Single dedicated ES-module worker owns the model + processor
  and runs decode → inference → mask upscale → alpha composite → PNG encode. Comlink
  for RPC. Main thread only paints UI.

## Privacy Model
- **Protected:** the user's image never leaves the tab (all decode / inference /
  composite / encode happen locally); no analytics, cookies, third-party fonts, or
  telemetry; no account or API key.
- **Not protected:** first-use only fetches the ~168 MB RMBG-1.4 ONNX weights from
  huggingface.co (HF's CDN sees an IP-level fetch, never the image); GitHub Pages /
  Cloudflare log the initial page-load like any site.
- **Trust surface:** the static bundle deployed by the GitHub Action pinned to its
  commit; the TLS chain to bgwipe.benrichardson.dev; the RMBG-1.4 weights; the
  transformers.js runtime.

## Architecture Decisions
- **Vanilla TS over React** — single workflow (input → process → output). No pane
  orchestration justifies the framework overhead.
- **RMBG-1.4 (briaai) over MODNet or ISNet** — RMBG-1.4 is trained on general
  subjects (products, people, animals) and matches the ecommerce/social audience.
  MODNet is portrait-focused. Licence permits personal/non-commercial use, which fits
  a free-forever open-source tool.
- **FP32 model over quantized** — WebGPU handles FP32 in ~1–2s per image, and the
  quality is noticeably better than Q8 for edge quality. The trade-off is a bigger
  first-load (~168 MB, not the 44 MB I initially estimated); corrected in copy.
- **Comlink** — RPC boundary reads like ordinary async calls, subscribe pattern for
  progress streaming is clean.
- **Light theme + sage accent + spacious layout** — matches the audience. The trust
  pill and privacy modal do the heavy lifting on the privacy narrative; the rest of
  the UI is calm.

## Errors & Resolutions
- **TS compile error: `workerRaw` unused** — removed the stored reference; worker is
  wrapped in the Comlink proxy at the callsite and doesn't need a raw handle.
- **TS compile error: `navigator.gpu` unknown** — narrowed the global cast to a
  local `{ gpu?: unknown }` shape instead of relying on the base `Navigator` type,
  which does not include the WebGPU addition in the current lib.dom.d.ts.
- **Model size was 168 MB not 44 MB** — RMBG-1.4 in FP32 (which WebGPU wants for
  best quality) is ~168 MB, not the ~44 MB quantized variant. Corrected the copy in
  index.html, README.md, and the in-app progress sub-line; pushed a fix commit to
  main and rebased the review branch.
- **TLS cert stuck at `authorization_created`** — Cycled the CNAME per procedure.
  Cert did not switch to `approved` within the poll window. This is a known GH Pages
  behaviour on first setup; the domain will become HTTPS-live automatically. HTTP
  redirect already works.
- **Repo pre-existed** — a previous run had scaffolded, deployed, and opened PR #3.
  Detected this state, updated the copy, cherry-picked to main, force-pushed the
  review branch on top of main. Did not create a duplicate PR.
