## Idea Source
Researched. IDEAS.md was empty. A `tools/`-vs-registry diff surfaced only the two known orphan duplicates (`collate`, `pagewell` — same job as the deployed `pagesmith`, from a concurrent-run collision), which per project memory are intentionally left alone, so there was no genuine backlog to finish.

Scanned the catalog for gaps: file conversion, compression, PDF, encryption, P2P, GIF/audio/screen capture, OCR, QR and JWT are all covered — but there was **no general file checksum / hash tool**. "Hash a file online" / "verify SHA-256 checksum" is a high-volume, evergreen query, and every existing tool asks you to upload the file, which is the exact anti-pattern for the sensitive/large files people actually verify. That's a strong privacy-first, client-side fit with a genuine WASM + Streams pillar.

## Tool Details
- **Name:** Verisum
- **Repo:** ben-gy/verisum
- **Category:** encryption-privacy (index category: `privacy`)
- **Audience:** developers, sysadmins, security-conscious downloaders verifying an ISO / installer / backup against a published checksum
- **Stack:** Vite 6 + vanilla TypeScript + Vitest
- **Browser APIs:** hash-wasm (WASM streaming), `File.stream()` ReadableStream, Web Workers, Web Crypto (SubtleCrypto/HMAC), Clipboard, Drag & Drop / File API, Service Worker (PWA)
- **Worker strategy:** single dedicated worker; streams each file via `File.stream()`, fans chunks into one incremental hasher per selected algorithm, posts progress + throughput, supports cooperative abort. Files processed sequentially.

## Privacy Model
- **Protected:** the file's bytes and any text/keys typed never leave the device — no upload endpoint exists in the code; fully offline after load.
- **Not protected:** a checksum proves integrity, not authenticity (a signature does that); MD5/SHA-1/CRC-32 are weak and offered only for corruption checks; the file itself is not a secret once you share it.
- **Trust surface:** hash-pinned GitHub Pages bundle + TLS; the bundled hash-wasm WASM; cookie-less Cloudflare Web Analytics page-view beacon (files never sent to it).

## Architecture Decisions
- **hash-wasm** chosen over SubtleCrypto because browsers can't natively do MD5, CRC-32, BLAKE2b/BLAKE3, SHA3, or xxHash, and SubtleCrypto.digest can't stream (it needs the whole buffer in memory). hash-wasm gives incremental `update()` hashers and bundles its WASM inline (base64) → no separate asset to fetch, works offline, satisfies CSP with `wasm-unsafe-eval`.
- **Streaming via `File.stream()`** keeps memory flat for multi-GB files — the whole point of a local verifier (you can't upload a 4 GB ISO, and you shouldn't load it fully into RAM either).
- **Vanilla TS** (no React): three simple tabs, no deep component tree.
- **Verify auto-detect:** expected checksum normalised (handles `algo:hex`, GNU `hash  file`, BSD `SHA (file) = hash`, `*binary` marker, uppercase), algorithm inferred by hex length, auto-enabled/computed, and equal-length algorithms (SHA-256 / SHA3-256 / BLAKE2b / BLAKE3 all 64 hex) disambiguated by actual value.

2. **"More algorithms" expander did nothing.** The `#algo-more` button was never given a click handler. Fix: wired it in `render()` to toggle `showAllAlgos` and re-render the chips (4 ↔ 9). Verified in-browser.
3. **`tsc` unused-import error** in `verify.ts` (`AlgoId` imported but unused). Fix: removed it from the type import.
